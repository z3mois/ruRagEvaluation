\section{Интерактивный сервис оценки}
\label{sec:service}

\subsection{Назначение}

Помимо обученных моделей и опубликованного датасета, 
практическая ценность работы включает интерактивный сервис,
 позволяющий прикладным пользователям оценивать свои RAG-ответы
 по TRACe-метрикам, не требуя написания кода. Сервис реализован на Streamlit 
 (модуль \texttt{new\_streamlit.py}) и запускается командой 
 \texttt{streamlit run new\_streamlit.py}; инференс работает на CPU или
  одном GPU без специальной конфигурации.

\subsection{Архитектура}

Сервис состоит из трёх логических блоков:

\begin{itemize}
  \item \textbf{Регистр моделей} (словарь \texttt{MODEL\_REGISTRY}). На момент 
  публикации доступно три эксплуатационные конфигурации:
    \begin{itemize}
      \item \texttt{DeBERTa-v3-large (len=512)} - контрольная базовая конфигурация;
      \item \texttt{ModernBERT (len=1024)} - основная русскоязычная модель;
      \item \texttt{BAAI-bge-m3 (len=1024)} - многоязычная модель для смешанных 
      корпусов.
    \end{itemize}
    Веса хранятся локально в \texttt{model\_utils/sweeps3/<model>/len\_<L>/model.pt} 
    и подгружаются через \texttt{@st.cache\_resource}, что позволяет держать в памяти 
    только одну модель за раз и мгновенно переключаться между уже использованными.
  \item \textbf{Модуль инференса.} Внутри сервиса определён самостоятельный класс
   \texttt{DebertaTrace} (базовая модель плюс три линейные проекции), совместимый по
    \texttt{state\_dict} с обучающей реализацией \texttt{DebertaTraceSimple} из
     \texttt{model\_utils/models.py}. Такое разделение позволяет подгружать чекпоинты
      из обучающего пайплайна без импорта самого пайплайна. На вход принимается
       тройка (вопрос, контекст, ответ) в формате RAGBench-RU.
  \item \textbf{Интерфейс пользователя.} Два основных режима:
    \begin{enumerate}
      \item \emph{Одиночный режим} (single-instance) - оценка одной тройки (вопрос, контекст, ответ)
      . Ввод через текстовые поля Streamlit, вывод - три TRACe-скора,
       сырые вероятности по токенам, подсветка релевантных и использованных предложений
        в тексте документа.
      \item \emph{Пакетный режим} (batch) - оценка CSV/JSONL-файла с 
      колонками \texttt{question\_ru}, \texttt{response\_ru}, 
      \texttt{documents\_sentences\_ru}. Результат возвращается как CSV с 
      колонками \texttt{relevance\_score}, \texttt{utilization\_score}, 
      \texttt{adherence\_score} и скачивается через \texttt{st.download\_button}.
    \end{enumerate}
\end{itemize}

\subsection{Ключевые возможности пользовательского интерфейса}

\begin{itemize}
  \item \textbf{Ползунки порогов.} Пороги $\tau_R, \tau_U, \tau_A$ выставлены в значения по 
  умолчанию 0.5, но пользователь может подстроить их через ползунки; результат визуализации 
  обновляется без перезапуска модели. Это полезно для анализа конкретных примеров и для 
  интерактивного подбора порогов на собственных данных.
  \item \textbf{Визуализация токенной разметки.} В одиночном режиме текст документа 
  отрисовывается с двумя цветовыми слоями: жёлтая подсветка - токены 
  с $\sigma(\ell^{(R)}) \ge \tau_R$ (релевантные), синяя рамка - токены
   с $\sigma(\ell^{(U)}) \ge \tau_U$ (использованные). 
   Токены ответа подсвечиваются зелёным, если adherence выше порога.
  \item \textbf{Совместимость с HuggingFace.} Бэкенд умеет загружать как 
  локальные \texttt{model.pt}, так и модели, лежащие на HF 
  (достаточно указать \texttt{repo id}). Это даёт возможность 
  в будущем распространять сервис вместе с моделями без необходимости
   скачивать веса вручную.
  \item \textbf{Проверка входного формата.} В пакетном режиме файл 
  валидируется на наличие обязательных полей \texttt{REQUIRED\_BATCH\_FIELDS}; 
  при ошибке показывается понятное сообщение с указанием недостающих колонок.
\end{itemize}

\subsection{Ограничения}

На текущем этапе сервис решает только задачи одиночной оценки и 
офлайн-обработки пакетов; интеграция в CI-контуры (webhook, REST-API) 
оставлена на будущее. Кроме того, сервис не содержит встроенного
 инструмента дообучения на пользовательских данных - это 
 осознанный дизайнерский выбор: обучение модели требует разметки,
  которая за рамками интерактивного интерфейса; для таких сценариев
   рекомендуется использовать обучающий конвейер \texttt{model\_utils/runner.py}.
    Формальных замеров производительности (задержка на пример, пропускная способность
     на GPU/CPU) в рамках работы не проводилось.
