# Оценка RAG-систем для русского языка

## 1. Проблема, актуальность, постановка задачи

### Проблема

RAG (Retrieval-Augmented Generation) стал дефакто стандартом для создания QA систем с интеграцией внутренних "знаний". Однако существующие методы оценки RAG-систем:

- **Непрозрачны**: end-to-end метрики (BLEU, ROUGE, BERTScore) не показывают, *почему* ответ плохой — проблема в ретривере или в генераторе?
- **Неструктурированы**: нет единого фреймворка, который бы оценивал одновременно качество поиска контекста и качество генерации ответа на уровне отдельных предложений/токенов.
- **Ориентированы на английский**: практически все бенчмарки и обученные модели-оценщики работают только с английским языком. Для русского языка инструментов нет.

### Актуальность

- RAG активно используется в проде (поисковые ассистенты, юридические системы, медицинские чат-боты), но качество русскоязычных RAG-систем сложно измерить.
- Статья RAGBench (Firas Jolha et al., 2024, arXiv:2407.11005) предложила метод оценки TRACe с токен-уровневыми метриками и показала, что файнтюненная модель-классификатор превосходит LLM-based оценку.
- Перенос этого подхода на русский язык — открытая задача: нужен перевод данных, адаптация моделей, проверка работоспособности на русском тексте.

### Задачи диссертации

| # | Задача | Описание |
|---|--------|----------|
| 1 | Создание русскоязычного бенчмарка | Перевод RAGBench (7 поддатасетов) на русский, публикация на HuggingFace |
| 2 | Воспроизведение статьи | Реализация разметки из статьи на основе LLM , проверка на Qwen2.5-72B для русского текста |
| 3 | Обучение моделей-классификаторов | Файнтюнинг 4 моделей (DeBERTa, BGE-M3, RuModernBERT, RoPEBert) для токен-уровневой классификации: relevance, utilization, adherence |
| 4 | Сравнительный анализ | Сравнение моделей по архитектурам, длинам контекста, весам лосса; сравнение с LLM-based подходом |
| 5 | Сервис для оценки | Streamlit-приложение для интерактивной оценки RAG-систем |

## 2. Схема экспериментов

### 2.1. Общая схема

```
RAGBench (EN) ──[перевод Qwen2.5-72B]──> RAGBench-RU (HF: CMCenjoyer/ragbench-ru)
                                              │
                    ┌─────────────────────────┼─────────────────────────┐
                    ▼                         ▼                         ▼
            LLM-based оценка          Модели-классификаторы       Сервис (Streamlit)
            (промпт из статьи)        (token-level classification)
            Qwen2.5-72B                       │
                                    ┌─────────┼─────────┬──────────┐
                                    ▼         ▼         ▼          ▼
                                 DeBERTa   BGE-M3   ModernBERT  RoPEBert
                                    │         │         │          │
                                    └─────────┴─────────┴──────────┘
                                              │
                                    Сетка гиперпараметров:
                                    - Длины контекста (128..8192)
                                    - Веса лосса (1:1:1 и 3:3:1)
                                    - Пороги (подбор на val)
```

### 2.2. Детали экспериментов с моделями

| Модель | Длины контекста | Примечание |
|--------|----------------|------------|
| DeBERTa-v3-large | 128, 256, 512 | Нет RoPE, максимум ~512 |
| BAAI/bge-m3 | 512, 1024, 2048, 4096, 8192 | Мультиязычная, до 8192 |
| Tochka-AI/ruRoPEBert-e5-base-2k | 256, 512, 1024, 2048 | Русская, RoPE до 2048 |
| deepvk/RuModernBERT-base | 512, 1024, 2048, 4096, 8192 | Русская, ModernBERT до 8192 |

**Архитектура модели оценщика TRACe**:
- Backbone (трансформер) -> 3 параллельных линейных головы (relevance, utilization, adherence)
- Вход: `[question] [SEP] [documents] [SEP] [response]`
- Маски: `context_mask` (документы), `response_mask` (ответ)
- Loss: взвешенная сумма BCE для трёх таргетов: relevance, utilization, adherence

**Варианты весов лосса**:
- 1:1:1 (равные веса) — базовый
- 3:3:1 (акцент на relevance и utilization) — гипотеза: adherence проще, не нужен большой вес

**Метрики**:
- Замерил Precision / Recall / F1 для каждого из трёх таргетов (на уровне токенов)
- Реализовал подбор оптимальных порогов на валидации (сетка 0.1..0.9, шаг=0.01)
- цифры приложу отдельно в сводной таблице

### 2.3. LLM-based оценка/разметка

- Промпт из оригинальной статьи: LLM получает документы + вопрос + ответ, возвращает JSON с разметкой каждого предложения(а в следствие и токена)
- Модель: Qwen2.5-72B-Instruct
- Метрики: overlap_ratio, miss_count, extra_count для relevant и utilized ключей
- Сэмплирование: 15% от каждого поддатасета
